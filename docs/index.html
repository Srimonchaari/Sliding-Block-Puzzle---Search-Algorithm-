Algorithm one, uninformed

Breadth-First Search

Checks every board one move away before trying anything two moves away.

04
How it moves through the tree
Start in a queuefirst in, first out
↓
Take the state waiting longest thinking
↓
Is it the target? Done, and it's the shortest path.
↓ no
Look at every neighbor boardskip ones already seen
↓
Add new ones to the back of the queue
Pseudocode
function BFS(start, target):
  queue <- [start]
  seen <- { start }
  expansions <- 0

  while queue is not empty:
    node <- queue.removeFront()
    if node == target:
      return path(node), expansions

    expansions <- expansions + 1
    for each neighbor in expand(node):
      if neighbor not in seen:
        seen.add(neighbor)
        queue.addBack(neighbor)
  • Always finds the shortest solution, because it checks every shorter option first
  • Its weakness is memory: it has to hold every board at the current level all at once
Algorithm two, informed

A* Search

Same idea as BFS, but it always tries the board that looks closest to the goal first.

05
How it decides what to try next
Start in a priority listranked by f(n), not by arrival order
↓
Take the board with the lowest f(n) thinking
↓
Is it the target? Done, and it's still the shortest path.
↓ no
Work out g, h, and f for each neighbor
↓
Keep it only if this route is cheaper than before
Pseudocode
function A_STAR(start, target):
  frontier <- [(f: h(start), start)]
  bestCost <- { start: 0 }
  expansions <- 0

  while frontier is not empty:
    node <- frontier.removeLowestF()
    if node == target:
      return path(node), expansions

    expansions <- expansions + 1
    for each neighbor in expand(node):
      cost <- bestCost[node] + 1
      if cost < bestCost.get(neighbor, infinity):
        bestCost[neighbor] <- cost
        f <- cost + heuristic(neighbor, target)
        frontier.add((f, neighbor))
The heuristic: Manhattan distance
  • For each tile, add up how many rows and columns it is from home, as if it could walk straight through the other tiles
  • Example: tile 7 belongs at row 2, col 1. Sitting at row 0, col 0, its distance is 2 + 1 = 3
  • It never overguesses, because a real move can only cost the same or more, tiles have to go around each other
  • That's why A* still finds the shortest solution, not just a fast one
function manhattan(board, target):
  total <- 0
  for each tile in board, skip blank:
    (r, c)   <- where it is now
    (gr, gc) <- where it belongs
    total += |r-gr| + |c-gc|
  return total
A quick check

Can every layout be solved?

No. Only half of all 8-puzzle layouts can reach the target at all.

06
The rule
  • Read the tiles left to right, top to bottom, skip the blank
  • Count pairs where a bigger number comes before a smaller one, these are inversions
  • If that count is even, it can be solved
Why that's true
  • Sliding a tile sideways never changes the inversion count
  • Sliding it up or down moves it past two others, so the count shifts by 2, always even
  • The target has zero inversions, so only even starting counts can ever reach it
Our start board
  • Reading 8 7 6 5 4 3 2 1 gives an even count
  • So it's solvable, which matches both algorithms finding an answer
  • Inversion parity rule: Princeton COS226 [5], MIT 6.1200J [6]

Discussion

What we took away from this

11
Both get it right
  • BFS and A* land on the same 28-move solution, A* just gets there with far less work
The heuristic does the thinking
  • A* on its own is just a strategy, Manhattan distance is what actually points it toward the goal
Memory is the real cost
  • BFS holding 185,000 states, not just its runtime, is what would break it on a bigger puzzle like the 15-puzzle
A note on the code
  • No search-algorithm libraries, the queue, the priority list, and both search loops are hand-written
  • A seen-set for BFS and a best-known-cost map for A* stop us from repeating work or looping
  • Both share the same board and move logic, so the comparison is only about the search strategy
